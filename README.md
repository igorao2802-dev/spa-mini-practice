## 🎤 Interview Corner

### 1. Что такое «однонаправленный поток данных» в React? Почему это удобно?

> **Однонаправленный поток данных** (one-way data flow) — принцип, при котором данные в React передаются **только сверху вниз** через `props`, а события сообщаются **снизу вверх** через функции-коллбеки.
>
> **Почему это удобно:**
>
> - 🔍 **Предсказуемость**: всегда понятно, откуда пришли данные и кто может их изменить
> - 🐛 **Проще отладка**: если баг в интерфейсе — смотрим в один «источник истины»
> - 🧱 **Чёткие границы**: компоненты не меняют чужие данные, только запрашивают изменения
>
> **Пример из проекта:**
>
> ```jsx
> // App.js (владелец данных)
> const [bookings, setBookings] = useState(INITIAL_BOOKINGS);
>
> // Данные идут вниз через props
> <BookingList items={filteredBookings} onDelete={handleDelete} />;
>
> // Событие идёт вверх через коллбек
> function handleDelete(id) {
>   setBookings((prev) => prev.filter((item) => item.id !== id));
> }
> ```

### 2. Что такое lifting state up? Приведите пример из Вашей практической работы.

> **Lifting state up** («подъём состояния») — паттерн, при котором общее состояние переносится в ближайшего общего родителя компонентов, которым эти данные нужны.
>
> **Когда применять:** когда два и более компонента должны отображать или изменять одни и те же данные.
>
> **Пример из проекта:**
>
> ```jsx
> // App.js — общий родитель хранит состояние фильтра
> const [filter, setFilter] = useState("Все");
>
> // Фильтрация вычисляется здесь (бизнес-логика рядом с данными)
> const filteredBookings = filter === "Все"
>   ? bookings
>   : bookings.filter(item => item.status === filter);
>
> // Данные и коллбек передаются вниз
> <StatusFilter activeFilter={filter} onFilterChange={setFilter} />
> <BookingList items={filteredBookings} onDelete={handleDelete} />
> ```
>
> `StatusFilter` и `BookingList` не хранят фильтр сами — они получают его через `props` и сообщают о изменениях через `onFilterChange`.

### 3. Как дочерний компонент сообщает родителю о событии? Почему нельзя изменить state родителя напрямую?

> Дочерний компонент сообщает о событии **через функцию-коллбек**, которую родитель передаёт ему в `props`.
>
> **Нельзя менять state родителя напрямую**, потому что:
>
> - 🔒 **Инкапсуляция**: только владелец state имеет право его изменять
> - ⚡ **Реактивность**: прямое присваивание не запустит перерисовку компонента
> - 🎯 **Предсказуемость**: если бы дети могли менять чужой state, отладка стала бы невозможной
>
> **Правильный паттерн:**
>
> ```jsx
> // Родитель (App.js)
> function handleAddItem(newItem) {
>   setBookings((prev) => [...prev, newItem]);
> }
> <BookingForm onAddItem={handleAddItem} />;
>
> // Дочерний компонент (BookingForm.jsx)
> function handleSubmit(e) {
>   e.preventDefault();
>   // ...валидация...
>   onAddItem(newBooking); // Сообщаем родителю о событии
> }
> ```

### 4. Что такое управляемый компонент (controlled component)? Зачем нужен атрибут value у input?

> **Управляемый компонент** — элемент формы, значение которого полностью контролируется через React `state`.
>
> **Роль атрибута `value`:** он связывает инпут с состоянием — поле отображает **только то, что записано в state**.
>
> **Как это работает:**
>
> 1. `value={text}` — инпут показывает значение из state
> 2. `onChange={(e) => setText(e.target.value)}` — при вводе обновляем state
> 3. React перерисовывает компонент → инпут получает новое `value` → пользователь видит изменение
>
> **Преимущества:**
>
> - ✅ Валидация в реальном времени
> - ✅ Программная очистка/изменение полей
> - ✅ Синхронизация нескольких полей
> - ✅ Единый источник правды
>
> **Пример:**
>
> ```jsx
> function BookingForm() {
>   const [roomName, setRoomName] = useState("");
>
>   return (
>     <input
>       value={roomName}
>       onChange={(e) => setRoomName(e.target.value)}
>       placeholder="Название помещения"
>     />
>   );
> }
> ```

### 5. Что такое паттерн «компонент-обёртка» (wrapper component)? Как работает children?

> **Компонент-обёртка** (wrapper) — компонент, который задаёт общий каркас (шапка, стили, структура), а содержимое получает через специальный `prop` — **`children`**.
>
> **`children`** — это всё, что написано **между открывающим и закрывающим тегом** при использовании компонента.
>
> **Пример:**
>
> ```jsx
> // Layout.jsx
> export default function Layout({ title, children }) {
>   return (
>     <div className="layout">
>       <header className="layout__header">
>         <h1>{title}</h1>
>       </header>
>       {/* ПОЧЕМУ children? Это механизм композиции: 
>           Layout задаёт структуру, содержимое вкладывается извне */}
>       <main className="layout__main">{children}</main>
>     </div>
>   );
> }
>
> // Использование в App.js
> <Layout title="🏢 Бронирование помещений">
>   <BookingForm onAddItem={handleAddItem} />
>   <StatusFilter activeFilter={filter} onFilterChange={setFilter} />
>   <BookingList items={filteredBookings} onDelete={handleDelete} />
> </Layout>;
> ```
>
> **Зачем это нужно:** не дублировать шапку и структуру на каждой странице, легко менять дизайн в одном месте.

### 6. Почему в React предпочтительна композиция, а не наследование?

> **Композиция** — сборка интерфейса из мелких независимых компонентов через вложение.  
> **Наследование** — создание иерархии классов, где дочерние расширяют родительские.
>
> **Почему React выбирает композицию:**
> | Критерий | Композиция | Наследование |
> |----------|-----------|--------------|
> | Гибкость | ✅ Можно комбинировать любые компоненты | ❌ Жёсткая иерархия |
> | Переиспользование | ✅ Компоненты независимы | ❌ Зависимость от базового класса |
> | Тестирование | ✅ Легко тестировать изолированно | ❌ Нужно учитывать наследие |
> | Масштабирование | ✅ Добавляем новые компоненты без изменения старых | ❌ Изменение базового класса ломает всех наследников |
>
> **Пример композиции в проекте:**
>
> ```jsx
> // Вместо наследования от "BaseCard"
> <Layout>
>   <section className="section">
>     <BookingForm onAddItem={handleAddItem} />
>   </section>
>   <section className="section">
>     <BookingList items={filteredBookings} />
>   </section>
> </Layout>
> ```

### 7. Зачем нужен useEffect для работы с localStorage? Что такое «побочный эффект» (side effect)?

> **`useEffect`** — хук для выполнения операций, которые **не связаны напрямую с рендерингом** (запросы к API, работа с DOM, `localStorage`).
>
> **«Побочный эффект» (side effect)** — любое действие, которое:
>
> - Взаимодействует с внешним миром (браузер, сервер, хранилище)
> - Не возвращает JSX
> - Может выполняться асинхронно
>
> **Почему `useEffect` для `localStorage`:**
>
> - 🔄 Выполняется **после** рендера, не блокируя отрисовку
> - 🎯 Запускается только при изменении зависимостей (`[bookings]`)
> - 🛡 Позволяет обработать ошибки через `try/catch`
>
> **Пример из проекта:**
>
> ```jsx
> // Инициализация с загрузкой из localStorage
> const [bookings, setBookings] = useState(() => {
>   try {
>     const saved = localStorage.getItem(STORAGE_KEY);
>     return saved ? JSON.parse(saved) : INITIAL_BOOKINGS;
>   } catch {
>     return INITIAL_BOOKINGS; // ПОЧЕМУ? Защита от повреждённых данных
>   }
> });
>
> // Сохранение при каждом изменении
> useEffect(() => {
>   try {
>     localStorage.setItem(STORAGE_KEY, JSON.stringify(bookings));
>   } catch {
>     console.warn("⚠️ Не удалось сохранить данные");
>   }
> }, [bookings]); // ПОЧЕМУ? Зависимость гарантирует синхронизацию
> ```

### Поток данных в приложении

```
                    ┌─────────────────┐
                    │   App.js        │
                    │ (единый source  │
                    │   of truth)     │
                    └────────┬────────┘
                             │
         ┌───────────────────┼───────────────────┐
         ▼                   ▼                   ▼
┌────────────────┐ ┌────────────────┐ ┌────────────────┐
│ BookingForm    │ │ StatusFilter   │ │ BookingList    │
│ onAddItem ↑    │ │ onFilterChange↑│ │ items ↓        │

                                               │
                                               ▼
                                    ┌────────────────┐
                                    │ BookingCard    │
                                    │ onDelete ↑     │
                                    └────────────────┘

📥 Данные идут ВНИЗ через props
📤 События идут ВВЕРХ через callback-функции
```

### Принцип декомпозиции

> Каждый компонент отвечает **за одну задачу**:
>
> - `BookingCard` — только отображение данных одной брони
> - `BookingForm` — только сбор и валидация данных формы
> - `StatusFilter` — только отображение кнопок фильтрации
> - `BookingList` — только рендеринг списка через `.map()`
> - `Layout` — только каркас страницы
>
> **Бизнес-логика** (фильтрация, добавление, удаление, `localStorage`) живёт в `App.js` — это обеспечивает **единый источник истины**.

## 🔗 Связь с курсовым проектом

### 🎯 Тема курсового проекта

> **«Система онлайн-бронирования помещений»** — SPA для резервирования конференц-залов, переговорных и коворкинг-зон с календарём занятости, валидацией конфликтов и личным кабинетом пользователя.

### 🔄 Как эта практика помогает курсовому проекту

| Навык из практики             | Применение в курсовом проекте                                                                  |
| ----------------------------- | ---------------------------------------------------------------------------------------------- |
| **Однонаправленный поток**    | `App → RoomList → RoomCard`: данные о помещениях идут вниз, события бронирования — вверх       |
| **Lifting state up**          | Состояние фильтра (дата, тип помещения) хранится в родителе, синхронизирует календарь и список |
| **Управляемые формы**         | Многошаговая форма бронирования с валидацией в реальном времени (`zod` + `react-hook-form`)    |
| **Композиция через children** | `Layout` с шапкой/подвалом оборачивает все страницы, `Modal` — формы подтверждения             |
| **localStorage**              | Сохранение черновиков бронирований и предпочтений пользователя между сессиями                  |
| **Комментарии «ПОЧЕМУ?»**     | Документация архитектурных решений для команды и будущей поддержки                             |
